Globe sphere, dark space background, and star field are unchanged. Country polygon highlight color switches from amber fill to white fill at 40% opacity. Trip pinpoint dots switch from amber to white.

**D-05: Redesign Scope ‚Äî All Screens**
Every user-facing surface: Globe home chrome, all bottom sheets, active trip UI, all buttons.

**D-06: Live Activity ‚Äî Dynamic Island + Lock Screen**
ActivityKit greenfield implementation. Dynamic Island compact (pulsing dot + elapsed) + expanded (recording label + stats row + location row). ## Summary

Phase 03.2 has two workstreams that must be planned in sequence-aware waves: (1) the design system swap ‚Äî replacing all `Color.Nomad.cream/warmCard/amber` tokens and `AppFont.title()/subheading()` Playfair references with the new black/white glass system across 26 Swift files ‚Äî and (2) a greenfield ActivityKit Live Activity target.

The design system workstream is mechanically straightforward because the API surface of `AppFont.*` and `Color.Nomad.*` is preserved ‚Äî call sites do not change. The entire change propagates through three files (`AppColors.swift`, `AppFont.swift`, `PanelGradient.swift`) replaced atomically, and then the visual layer of each component/sheet is updated to apply the glass surface recipe. The critical subtlety is that `panelGlassSurface()` (replacing `panelGradient()`) requires a `.presentationBackground(Color.Nomad.panelBlack)` companion modifier on sheets, because SwiftUI's sheet container provides its own background that would otherwise show through.

The Live Activity workstream is greenfield and requires an Xcode Widget Extension target added to the project. The app's deployment target is already iOS 18.0 ‚Äî well above the ActivityKit minimum of iOS 16.2 ‚Äî so no version gating is needed. `NSSupportsLiveActivities` must be added to the main app's `Info.plist`. `TripActivityAttributes` must be added to both the main app target AND the widget extension target membership (no shared framework needed ‚Äî duplicate the single file). The `LocationManager` currently exposes `isRecording`, `currentTripId`, and writes `RoutePoint` records; it will need two new published properties (`accumulatedDistanceMeters: Double` and `lastLocation: CLLocation?`) plus a hook to compute distance incrementally and trigger Live Activity updates.

**Primary recommendation:** Plan the design system replacement as Wave 1 (token files ‚Üí component files ‚Üí sheet files ‚Üí globe chrome) and the Live Activity as Wave 2 (Xcode target + attributes struct + widget UI + LocationManager extension + GlobeView wiring). The two workstreams share no code dependencies in either direction.

### Pattern 1: Glass Surface ViewModifier (replacing panelGradient)

**What:** Drop-in replacement for `.panelGradient()` that produces the 4-layer dark glass surface.
**When to use:** On all bottom sheet ScrollView or VStack roots; replaces `.panelGradient()` at all 5 call sites.

```swift
// Source: UI-SPEC Glassmorphic Surface Contract
struct PanelGlassSurfaceModifier: ViewModifier {
    func body(content: Content) -> some View {
        content
            .background(
                ZStack {
                    Color.Nomad.panelBlack              // Layer 1: opaque base
                    Rectangle()                         // Layer 2: material blur
                        .fill(.ultraThinMaterial)
                    Color.black.opacity(0.35)           // Layer 3: darkness reinforcement
                }
            )
            .overlay(
                UnevenRoundedRectangle(
                    topLeadingRadius: 20,
                    bottomLeadingRadius: 0,
                    bottomTrailingRadius: 0,
                    topTrailingRadius: 20
                )
                .stroke(Color.white.opacity(0.20), lineWidth: 1) // Layer 4: hairline border
            )
            .clipShape(
                UnevenRoundedRectangle(
                    topLeadingRadius: 20,
                    bottomLeadingRadius: 0,
                    bottomTrailingRadius: 0,
                    topTrailingRadius: 20
                )
            )
    }
}

extension View {
    func panelGlassSurface() -> some View {
        modifier(PanelGlassSurfaceModifier())
    }
}
```

**Critical companion modifier on sheet call sites:**
```swift
// Source: UI-SPEC Sheet Presentation Contract
.presentationBackground(Color.Nomad.panelBlack)
```
Without this, SwiftUI's sheet container shows its own (light) background through the glass layers. [ASSUMED ‚Äî based on `.presentationBackground` documentation behavior; verify by inspection during execution]

import ActivityKit

func startLiveActivity(tripStartDate: Date) throws -> Activity<TripActivityAttributes>? {
    guard ActivityAuthorizationInfo().areActivitiesEnabled else { return nil }

### Pattern 8: LocationManager Extension for Live Activity Data

The current `LocationManager` does not expose `accumulatedDistanceMeters` or `lastLocation`. These must be added:

```swift
// Add to LocationManager.swift
var accumulatedDistanceMeters: Double = 0
var lastLocation: CLLocation?

// In saveRoutePoint(), after saving to SwiftData:
private func updateDistanceAndLocation(location: CLLocation) {
    if let last = lastLocation {
        accumulatedDistanceMeters += location.distance(from: last)
    }
    lastLocation = location
}
```

A periodic `Task` (every 30s) in GlobeView or a dedicated `TripActivityManager` reads these values and calls `updateLiveActivity`.

### Anti-Patterns to Avoid

- **Hardcoding colors inline instead of using Color.Nomad tokens:** After token replacement, new `warmCard`/`cream`/`amber` usages must not appear. Use `Color.Nomad.textPrimary`, `.textSecondary`, `.panelBlack`.
- **Forgetting `.presentationBackground(Color.Nomad.panelBlack)` on sheet presentations:** Without this, SwiftUI renders the system sheet container background (white/gray) beneath the glass layers, ruining the effect.
- **Adding `TripActivityAttributes.swift` to only one target:** The widget extension cannot render if it cannot see the attributes type. The file must be in both the app and widget extension target membership.
- **Calling `Activity.request()` without ending existing activities first:** Only one activity per `ActivityAttributes` type is recommended. Always iterate `Activity<TripActivityAttributes>.activities` and end them before starting a new one.
- **Using UIBlurEffect directly instead of `.ultraThinMaterial`:** The SwiftUI `.ultraThinMaterial` material automatically adapts to dark/light environments when used over a dark base ‚Äî the same behavior is desired. UIKit version is not needed unless finer-grained control is required.
- **Not guarding `ActivityAuthorizationInfo().areActivitiesEnabled`:** User can disable Live Activities per-app in Settings. Always check before calling `Activity.request()`.
- **Animating the pulsing dot in the widget:** WidgetKit Live Activity views run in a separate process and do not support arbitrary SwiftUI `@State`-driven animations. For the pulsing dot in the widget, use a `TimelineView` or omit the pulse animation in the widget (static dot is acceptable in WidgetKit context). ## Common Pitfalls

### Pitfall 1: Sheet Background Shows System Light Background Through Glass

**What goes wrong:** The glass surface renders but a white or light gray halo is visible behind it ‚Äî especially noticeable when the sheet is partially expanded.
**Why it happens:** SwiftUI's sheet container provides its own background (`systemBackground` or similar) that sits behind the view's content background. The glass layers are applied to the content but the container background shows through at sheet edges.
**How to avoid:** Add `.presentationBackground(Color.Nomad.panelBlack)` to every `.sheet(...)` modifier. This replaces the system container background with panelBlack before the glass layers composite on top.
**Warning signs:** Glass looks correct in Previews but has a light halo on device.

### Pitfall 2: .ultraThinMaterial Appears Too Light in Dark Context

**What goes wrong:** `.ultraThinMaterial` produces a gray/translucent surface that looks washed out rather than dark glass.
**Why it happens:** SwiftUI `.ultraThinMaterial` adapts to the environment color scheme. If the environment is light mode (e.g., during Previews without explicit dark scheme), the material appears light. On device, the globe is dark, but the sheet container may not propagate the dark scheme into the presented sheet.
**How to avoid:** Add `Color.black.opacity(0.35)` as a reinforcement layer over the material (already in the UI-SPEC recipe). Also set `.preferredColorScheme(.dark)` on sheets if needed. [ASSUMED ‚Äî verify by inspection on device]
**Warning signs:** Glass looks correct on dark backgrounds but pale/gray in Previews or light mode.

### Pitfall 3: PulseAnimation Not Working in Widget Extension

**What goes wrong:** The pulsing recording dot in the Dynamic Island / Lock Screen does not animate ‚Äî it renders as a static dot.
**Why it happens:** WidgetKit views run in a separate sandboxed process. Standard SwiftUI `@State`-driven animations driven by `onAppear` do not work in this context. WidgetKit supports only `TimelineView`-based transitions and limited `Animation` API. [ASSUMED ‚Äî based on WidgetKit animation constraint training knowledge]
**How to avoid:** In the widget extension, render the dot as a static white circle (no pulse). The pulsing animation in the in-app `RecordingPill` continues to work normally. Or use `TimelineView(.animation(minimumInterval: 0.8))` with a `sin`-based opacity ‚Äî but this is complex and the static dot is acceptable.
**Warning signs:** Simulator shows no animation on the Dynamic Island dot.

### Pitfall 4: TripActivityAttributes File Missing From Widget Extension Target

**What goes wrong:** Widget extension fails to compile with "Cannot find type 'TripActivityAttributes' in scope."
**Why it happens:** Xcode file target membership defaults to main app only. The widget extension is a separate target and needs explicit membership.
**How to avoid:** When creating `TripActivityAttributes.swift`, open the File Inspector (right panel in Xcode) ‚Üí Target Membership ‚Üí check both "Nomad" (main app) AND "NomadLiveActivity" (widget extension). No shared framework needed.
**Warning signs:** Widget extension build fails; main app builds fine.

### Pitfall 5: CLGeocoder Called Too Frequently from LocationManager

**What goes wrong:** App hits CLGeocoder rate limit, geocode requests fail silently, locationName stays as the first result forever.
**Why it happens:** CLGeocoder documentation states one request at a time per process; rapid calls during recording cause queued requests to fail.
**How to avoid:** Throttle geocoding: maintain `lastGeocodeDate: Date?` and only call if `Date().timeIntervalSince(lastGeocodeDate ?? .distantPast) >= 60`. Cache the last successful result. If geocode fails, keep the previous `locationName`.
**Warning signs:** locationName always shows "Locating..." or never updates after first result.

### Pitfall 6: Onboarding Screens Outside Redesign Scope

**What goes wrong:** Build succeeds but Onboarding screens (7 files, 35 amber/cream usages) still show the old amber palette ‚Äî creates a jarring visual inconsistency.
**Why it happens:** CONTEXT.md D-05 scope says "every user-facing surface" but the canonical refs list only targets the main post-onboarding flow. Onboarding screens also use `Color.Nomad.amber` and `Color.Nomad.cream` heavily.
**How to avoid:** The planner must decide whether to include Onboarding screens in scope. Research finding: 35 old-token usages exist across 7 onboarding files. If included, they follow the same token substitution pattern (cream ‚Üí panelBlack, amber ‚Üí textPrimary/accent). This research recommends including them ‚Äî amber will remain visible at first launch otherwise.
**Warning signs:** App looks correct after login but amber/cream is visible during onboarding flow.

### Pitfall 7: Removing Playfair Font Assets Breaks Font Assertions in AppFont DEBUG Block

**What goes wrong:** The `#if DEBUG` block in `AppFont.swift` contains `assert(UIFont(name: "PlayfairDisplay-Regular", size: 20) != nil)` which will crash in DEBUG builds after Playfair fonts are removed from the bundle.
**How to avoid:** The UI-SPEC explicitly calls this out ‚Äî update assertions from `PlayfairDisplay-*` to `Inter-SemiBold`. [VERIFIED: read AppFont.swift lines 39-40]
**Warning signs:** DEBUG builds crash on app launch; release builds unaffected.

import ActivityKit

func startTripLiveActivity() throws -> Activity<TripActivityAttributes>? {
    guard ActivityAuthorizationInfo().areActivitiesEnabled else { return nil }

            distanceKm: 0,
            elapsedSeconds: 0,

            isRecording: true

    )

        attributes: TripActivityAttributes(),
        content: content,
        pushType: nil
    )
}

## Open Questions (RESOLVED)

1. **Onboarding Screen Scope**
   - What we know: D-05 says "every user-facing surface". 7 onboarding files have 35 old token usages.
   - What's unclear: Whether the user intends onboarding screens to be in this phase or deferred (onboarding is rarely seen after first launch).
   - Recommendation: Include onboarding screens in Wave 1 token sweep. They use the same substitution pattern as all other files. Leaving them out creates visible inconsistency on first launch.
   - RESOLVED: Included in Plan 02 Task 3 per D-05 ("every user-facing surface" scope)

2. **TripActivityManager vs. Inline in GlobeView**
   - What we know: GlobeView.swift already manages trip lifecycle (start/stop, HealthKit query, TripService finalization). Adding Live Activity calls here is low-risk.
   - What's unclear: Whether a dedicated `TripActivityManager.swift` class would be cleaner for testability.
   - Recommendation: Add Live Activity calls inline in GlobeView for v1 (consistent with existing trip lifecycle pattern). Extract to manager if complexity grows.
   - RESOLVED: Inline in GlobeView for v1 (simpler, no extra type needed)

3. **LocationManager Distance Accumulation**
   - What we know: LocationManager currently only saves RoutePoints to SwiftData. It does not expose accumulated distance or last location for in-flight reads.
   - What's unclear: Whether the Live Activity should read accumulated distance from SwiftData query (heavier) or from an in-memory computed property on LocationManager (lighter).
   - Recommendation: Add `accumulatedDistanceMeters: Double` and `lastLocation: CLLocation?` as `@Observable` published properties to LocationManager, computed incrementally in `saveRoutePoint()`. This avoids SwiftData queries in the background update loop.
   - RESOLVED: In-memory `accumulatedDistanceMeters` incremented in `saveRoutePoint()`, fed to activity.update() calls in GlobeView
