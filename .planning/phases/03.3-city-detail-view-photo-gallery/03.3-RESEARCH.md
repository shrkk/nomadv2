## Summary

Phase 03.3 is a targeted enhancement to the existing `CountryDetailSheet`. The core architectural move is replacing the current single-level paged TabView (one page per city, one photo per page) with a two-level nested TabView structure: an outer TabView pages between cities (D-08/D-09), and an inner TabView within each city page scrolls through all that city's photos (D-02).

The UI-SPEC has made all visual decisions: the outer city TabView wraps the entire content area below the country header (city strip + photos + stats + logs all move together as one page unit). The inner photo TabView is nested inside each city page and uses a local `@State var photoIndex` that resets to 0 on city change. The `selectedCityIndex` binding on `CountryDetailViewModel` remains the single source of truth linking the outer TabView, city strip highlight, and city strip tap gestures.

The ViewModel needs one significant change: `loadCluster()` currently stores `photos: [UUID: UIImage]` (one image per cluster). This must change to `cityPhotos: [UUID: [UIImage]]` (all images per cluster) and add GPS bounding box filtering to match `PhotoGalleryStrip`'s established pattern. All other components — `StatsPillRow`, `TripLogCard`, `CityThumbnailCard`, `TemperatureNotchPill` — are reused without modification.

**Primary recommendation:** Implement the outer city TabView directly inside `CountryDetailSheet.body` (replacing the current flat ScrollView layout), extract city-page content into a `CityDetailPage` view that receives a single `CityCluster`, and update `CountryDetailViewModel` to load all photos per cluster using the bounding-box pattern from `PhotoGalleryStrip`.

### Pattern 1: Outer City TabView Replacing Flat ScrollView

**What:** The current `CountryDetailSheet.body` uses a single `ScrollView(.vertical)` with a flat `VStack`. Replace the section below the country header with a `TabView(selection: $viewModel.selectedCityIndex)` that pages between `CityDetailPage` instances (one per cluster). Each page contains the city strip + photo carousel + stats + logs as a vertically-scrollable unit.

**When to use:** D-08 requires full-page city swipe. The UI-SPEC mandates the city strip scrolls with the page (non-sticky), so the entire page including the strip must be the TabView content.

**How the current code maps to the new structure:**

Current layout (flat VStack inside ScrollView):
```
ScrollView(.vertical)
  VStack
    countryHeaderRow          ← STAYS outside TabView (persistent, D-04)
    cityStripSection          ← MOVES inside TabView per page
    CityPhotoCarousel         ← MOVES inside TabView per page (restructured)
    StatsPillRow              ← MOVES inside TabView per page
    tripLogsSection           ← MOVES inside TabView per page
```

New layout:
```
VStack(spacing: 0)
  countryHeaderRow            ← Outside TabView (persistent)
  TabView(selection: $viewModel.selectedCityIndex)
    ForEach clusters
      CityDetailPage(cluster:, viewModel:, onTripTap:)  ← One page per city
        ScrollView(.vertical)
          VStack
            cityStripSection  ← Repeated per page (same clusters, different selectedIndex)
            MultiPhotoCarousel(cluster:, photos:, temperature:)
            StatsPillRow(...)
            tripLogsSection(cluster:)
  .tabViewStyle(.page(indexDisplayMode: .never))
  .frame(maxHeight: .infinity)
```

**Critical note:** The country header stays outside and above the TabView — it is not part of the paged content (D-04). The `.frame(maxHeight: .infinity)` on the TabView allows it to fill remaining height so the outer container is a `VStack` not a `ScrollView`.

### Pattern 2: Inner Photo TabView in CityPhotoCarousel

**What:** `CityPhotoCarousel` currently takes `photos: [UUID: UIImage]` (one image per cluster). Modify it to accept `photos: [UUID: [UIImage]]` and render an inner `TabView` with a local `@State var photoIndex: Int = 0` for the selected photo within a city.

**When to use:** D-02 requires all photos per city, swipeable.

**Key implementation detail — photoIndex reset:** When the city changes (outer TabView swipes), the inner `photoIndex` must reset to 0. Since `CityPhotoCarousel` now renders a single city's photos (not all cities), this reset happens automatically — each city page gets its own `CityPhotoCarousel` instance with its own `@State var photoIndex`, and SwiftUI creates a fresh instance per TabView page.

**Photo count gate:** Per UI-SPEC, the page indicator dot row shows only when the city has 2+ photos. Gate with `if photos.count >= 2`.

**Example (inner carousel structure):**
```swift
// Source: UI-SPEC Multi-Photo Carousel spec + existing CityPhotoCarousel.swift pattern
struct MultiPhotoCarousel: View {
    let cluster: CityCluster
    let photos: [UIImage]        // All photos for THIS city
    let temperature: String?

    @State private var photoIndex: Int = 0

    var body: some View {
        VStack(alignment: .leading, spacing: 0) {
            ZStack(alignment: .top) {
                TabView(selection: $photoIndex) {
                    if photos.isEmpty {
                        emptyPhotoPlaceholder
                            .tag(0)
                    } else {
                        ForEach(Array(photos.enumerated()), id: \.offset) { index, image in
                            Image(uiImage: image)
                                .resizable()
                                .scaledToFill()
                                .frame(maxWidth: .infinity)
                                .frame(height: 320)
                                .clipped()
                                .tag(index)
                        }
                    }
                }
                .tabViewStyle(.page(indexDisplayMode: .never))
                .frame(height: 320)
                .clipShape(RoundedRectangle(cornerRadius: 16))

                // Temperature notch overlay
                if let temp = temperature {
                    TemperatureNotchPill(temperature: temp)
                        .padding(.top, 12)
                }
            }

            // Photo page dots — only if 2+ photos
            if photos.count >= 2 {
                photoPageIndicator
                    .padding(.top, 8)
            }

            locationIdentityBlock
        }
    }
}
```

### Pattern 3: ViewModel — Load All Photos Per Cluster

**What:** `CountryDetailViewModel.loadCluster()` currently:
1. Fetches `fetchLimit = 1` (one representative photo) → stored in `photos: [UUID: UIImage]`
2. Counts all photos → stored in `photoCountPerCluster: [UUID: Int]`

This phase requires:
1. Fetch ALL photos for the cluster date range + GPS bounding box → stored in `cityPhotos: [UUID: [UIImage]]`
2. `photoCountPerCluster` stays but is now derived from `cityPhotos[id]?.count`

**GPS bounding box filter:** `PhotoGalleryStrip` already implements the bounding box pattern. Adapt it for clusters. The bounding box must be computed from all `trip.routePreview` coordinates in the cluster:

```swift
// Source: PhotoGalleryStrip.swift bounding box pattern — adapted for CityCluster
private func boundingBox(for cluster: CityCluster) -> (minLat: Double, maxLat: Double, minLon: Double, maxLon: Double)? {
    let coords = cluster.trips.flatMap { $0.routePreview }
    guard !coords.isEmpty else { return nil }
    let lats = coords.map { $0[0] }
    let lons = coords.map { $0[1] }
    return (lats.min()!, lats.max()!, lons.min()!, lons.max()!)
}
```

**Fetch all photos (no fetchLimit), load with `.opportunistic` delivery:**
```swift
// Load degraded frame immediately for placeholder, replace on final
let reqOptions = PHImageRequestOptions()
reqOptions.deliveryMode = .opportunistic   // degraded first, then high quality
reqOptions.isSynchronous = false
reqOptions.isNetworkAccessAllowed = true
```

**Memory cap:** Per UI-SPEC, evict full-res images for clusters more than 2 steps from `selectedCityIndex`. Implement in `CountryDetailViewModel` as a computed set of "active" cluster IDs (visible + adjacent), clearing `cityPhotos` entries for distant clusters.

[VERIFIED: CountryDetailViewModel.loadCluster() — current implementation; PhotoGalleryStrip.swift — bounding box pattern]

### Pattern 4: selectedCityIndex as Single Source of Truth

**What:** `CountryDetailViewModel.selectedCityIndex: Int` already drives:
- `CityPhotoCarousel` outer TabView selection
- City strip `isSelected` state per `CityThumbnailCard`

In Phase 03.3, the outer city TabView (`selection: $viewModel.selectedCityIndex`) directly binds to this same property. No additional state is needed — the TabView binding and city strip tap handler both write to `viewModel.selectedCityIndex`.

**Animation on tap:** When the user taps a city thumbnail card, wrap the assignment in `withAnimation`:
```swift
// Source: UI-SPEC Interaction Contract — spring response 0.35, damping 0.85
.onTapGesture {
    withAnimation(.spring(response: 0.35, dampingFraction: 0.85)) {
        viewModel.selectedCityIndex = index
    }
}
```

The outer TabView swipe updates `selectedCityIndex` automatically via its binding — no explicit animation needed for swipe-driven changes (system handles it).

[VERIFIED: CountryDetailSheet.swift cityStripSection — existing onTapGesture pattern]

### Anti-Patterns to Avoid

- **Sticky city strip:** The UI-SPEC explicitly calls for a non-sticky city strip (simpler non-pinned layout). Do not use `LazyVStack` with `pinnedViews` or a separate top overlay — the city strip scrolls with the page.
- **Shared photoIndex across cities:** Never use `viewModel.selectedPhotoIndex` as a shared binding for the inner carousel. Each city page must own its photo index locally (`@State private var photoIndex`) so navigating away and back resets to first photo.
- **fetchLimit on loadCluster:** Remove `options.fetchLimit = 1` — it blocks loading multiple photos.
- **Blocking the main thread with PHImageManager:** Keep `withCheckedContinuation` pattern from existing `loadCluster` — never call `isSynchronous = true` for carousel photos.
- **Nesting ScrollView inside TabView without height:** A `ScrollView` inside a TabView page needs a `.frame(maxHeight: .infinity)` or a known height to participate correctly in the paged layout — without it, the page height collapses.
- **Outer TabView inside ScrollView:** The outer city TabView cannot be inside a `ScrollView(.vertical)` at the same level — it will conflict. The new layout switches to `VStack` as the root container (country header + TabView), not a wrapping ScrollView. Vertical scroll lives inside each TabView page.

## Common Pitfalls

### Pitfall 1: Outer TabView + Inner TabView Gesture Conflict

**What goes wrong:** Nested horizontal TabViews (outer = cities, inner = photos) can intercept each other's swipe gestures, making both unresponsive or fighting for the touch.

**Why it happens:** Both TabViews want horizontal swipe. SwiftUI's gesture disambiguation may fail when both have equal priority.

**How to avoid:** The outer city TabView should wrap the full page including the inner carousel. SwiftUI's `TabView(.page)` nesting works correctly when the inner TabView is sized to a fixed height (`frame(height: 320)`) and the outer TabView's page is a `ScrollView`. System disambiguation: a horizontal swipe starting on the inner carousel is handled by the inner TabView; a horizontal swipe starting outside the carousel (or as an edge gesture) is handled by the outer TabView. Do not add custom `DragGesture` modifiers that compete with system tab paging — this is the fastest path to conflict.

**Warning signs:** Inner photo swipe navigates cities instead of photos; tapping the outer edge of the carousel triggers city navigation unexpectedly.

[ASSUMED — based on SwiftUI nested TabView behavior patterns; verify during implementation smoke test]

### Pitfall 2: TabView Height Collapse Inside VStack

**What goes wrong:** The outer city TabView renders with zero height or clips content.

**Why it happens:** When placed in a `VStack` without an explicit frame, `TabView` may not expand to fill available space because it has no intrinsic content size in paged mode.

**How to avoid:** Apply `.frame(maxHeight: .infinity)` to the outer city TabView so it fills the remaining space below the country header. Each page's inner `ScrollView` then scrolls within that allocated height.

**Warning signs:** Sheet shows country header and nothing below it; or only a sliver of the carousel is visible.

[VERIFIED: known SwiftUI TabView sizing behavior — consistent with existing CityPhotoCarousel usage which uses explicit `.frame(height: 320)` on the inner TabView]

### Pitfall 3: photos Dictionary Key Mismatch After ViewModel Rename

**What goes wrong:** Switching `photos: [UUID: UIImage]` to `cityPhotos: [UUID: [UIImage]]` requires updating every call site. Missing a call site silently renders an empty carousel without a compile error (the old dict still compiles if not deleted).

**Why it happens:** Swift dictionaries return `nil` for missing keys — no crash, just an empty array.

**How to avoid:** Delete `photos: [UUID: UIImage]` from `CountryDetailViewModel` entirely (don't shadow it). Let the compiler surface every call site. Update `cityStripSection` thumbnail source and `StatsPillRow` `photoCount` to use `cityPhotos[id]?.count ?? 0`.

**Warning signs:** City thumbnails show placeholder even after loading completes; stats pill shows 0 photos despite having loaded them.

[VERIFIED: CountryDetailSheet.swift — confirmed `photos` dict is referenced in cityStripSection and CityPhotoCarousel]

### Pitfall 4: Temperature Key Semantics (per cluster, not per photo)

**What goes wrong:** Implementing the temperature overlay as if each photo has its own temperature — creating N weather fetches per city.

**Why it happens:** The inner TabView has one photo per page, making it tempting to associate state per photo.

**How to avoid:** Temperature is still per-cluster (one value per city). The `TemperatureNotchPill` overlays on every photo slide within a city — all using the same cluster temperature. Pass temperature to `MultiPhotoCarousel` as a single `String?` parameter, same as the current `temperatureOverlay(for: cluster)` pattern.

[VERIFIED: CityPhotoCarousel.swift temperatureOverlay — current per-cluster temperature pattern]

### Pitfall 5: Memory Pressure from Loading All Photos for All Cities

**What goes wrong:** Loading all high-res photos for all clusters upfront causes memory warnings and eventual app termination on devices with limited RAM.

**Why it happens:** A country with 10 cities × 30 photos each = 300 full-res UIImages in memory simultaneously.

**How to avoid:** Per UI-SPEC loading strategy: load photos for visible city + immediately adjacent cities (prev + next). Evict `cityPhotos[id]` entries for clusters more than 2 positions away from `selectedCityIndex` when the index changes. Use `.opportunistic` delivery mode — degraded thumbnails load immediately while high-quality loads in background.

**Warning signs:** `os_log` memory warnings; app crash after swiping through 4-5 cities on older devices.

[ASSUMED — standard iOS PHAsset memory management practice; eviction threshold of 2 is from UI-SPEC recommendation]

### Resumed-Flag Pattern for PHImageManager (Existing, Reuse)
```swift
// Source: CountryDetailSheet.swift CountryDetailViewModel.loadCluster()
// Use this exact pattern for loading each photo in the multi-photo fetch
await withCheckedContinuation { cont in
    var resumed = false
    let reqOptions = PHImageRequestOptions()
    reqOptions.deliveryMode = .opportunistic  // change from .highQualityFormat for phase 3.3
    reqOptions.isSynchronous = false
    reqOptions.isNetworkAccessAllowed = true

    PHImageManager.default().requestImage(
        for: asset,
        targetSize: targetSize,
        contentMode: .aspectFill,
        options: reqOptions
    ) { image, info in
        let isDegraded = (info?[PHImageResultIsDegradedKey] as? Bool) ?? false
        if !isDegraded && !resumed {
            resumed = true
            cont.resume(returning: image)
        }
    }
}
```

### GPS Bounding Box from RoutePreview (Adapted from PhotoGalleryStrip)
```swift
// Source: PhotoGalleryStrip.swift bounding box logic — adapted for CityCluster
private func boundingBox(for cluster: CityCluster) -> (minLat: Double, maxLat: Double, minLon: Double, maxLon: Double)? {
    // routePreview entries are [[lat, lon], ...]
    let coords = cluster.trips.flatMap { $0.routePreview }
    guard !coords.isEmpty else { return nil }
    let lats = coords.compactMap { $0.first }
    let lons = coords.compactMap { $0.last }
    guard !lats.isEmpty, !lons.isEmpty else { return nil }
    return (lats.min()!, lats.max()!, lons.min()!, lons.max()!)
}
```
