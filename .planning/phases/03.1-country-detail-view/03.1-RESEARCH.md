## Summary

Phase 03.1 adds a Country Detail Sheet that slides up when a user taps a highlighted country on the globe. The sheet surfaces per-city trip data: a horizontal city thumbnail strip (proximity-clustered), a full-width paged photo carousel with temperature notch, location identity block, stats pill, and trip log entries. All data comes from existing Firestore trip documents already loaded in `GlobeViewModel.trips` ‚Äî no new data sources are required beyond WeatherKit for current temperature.

The existing codebase provides the complete scaffolding needed: `GlobeViewModel` already holds all `TripDocument` records and `visitedCountryCodes`; `TripDocument` has `cityName`, `distanceMeters`, `routePreview`, `startDate`, `endDate`, and `visitedCountryCodes`; `ProfileSheet` demonstrates the nested `.sheet()` pattern (INFRA-02) that this phase must replicate; `PhotoGalleryStrip` demonstrates PHAsset loading with authorization checking. The main new technical problems are: (1) grouping trips by geographic proximity to form "cities", (2) loading a single representative PHAsset per city for the carousel, and (3) integrating WeatherKit with entitlement and fallback.

**Primary recommendation:** Wire `GlobeViewModel.animateToCountry(code:)` to present `CountryDetailSheet` (a new `.sheet`) instead of (or before) showing ProfileSheet. The CountryDetailSheet receives the country code and the full `trips` array, filters to trips for that country, clusters them by proximity, and drives all child views from that local state.

### Pattern 1: INFRA-02 Nested Sheet (Critical)

**What:** CountryDetailSheet presented as a `.sheet` nested inside `GlobeView` (attached to `GlobeMapView` or the ZStack). TripDetailSheet is then nested inside `CountryDetailSheet` ‚Äî not alongside it. This is the same INFRA-02 pattern already proven in ProfileSheet.

**When to use:** Any time a second-level sheet needs to slide up over a first-level sheet without cascading dismiss.

**Example (from existing ProfileSheet):**
```swift
// Source: Nomad/Sheets/ProfileSheet.swift ‚Äî the INFRA-02 pattern
// TripDetailSheet nested INSIDE ProfileSheet body:
.sheet(isPresented: $showTripDetail) {
    if let trip = detailTrip {
        TripDetailSheet(trip: trip)
    }
}
```

CountryDetailSheet must follow the same nesting pattern for TripDetailSheet:
```swift
// Inside CountryDetailSheet body:
.sheet(isPresented: $showTripDetail) {
    if let trip = selectedTripForDetail {
        TripDetailSheet(trip: trip)
    }
}
```

**Critical:** The new `CountryDetailSheet` sheet must be attached to `GlobeMapView` (or its container) in `GlobeView`, NOT as a sibling of the existing `ProfileSheet`. If it is a sibling `.sheet`, only one can present at a time on iOS ‚Äî the second call silently wins and replaces the first. [VERIFIED: existing ProfileSheet code; INFRA-02 spike in REQUIREMENTS.md]

### Pattern 2: Proximity-Based City Clustering

**What:** Group `TripDocument` records for a given country by geographic proximity. The "city" grouping is derived by clustering trip start coordinates ‚Äî trips within `N` km of each other belong to the same city group.

**When to use:** Building the city strip. A country with 5 trips to Tokyo and 3 trips to Kyoto should produce 2 city groups, not 8 separate entries.

**Recommended approach ‚Äî greedy radius clustering:**
```swift
// Source: [ASSUMED] ‚Äî standard greedy centroid clustering, no external library needed
struct CityCluster {
    let cityName: String          // Most common cityName among member trips
    let centroid: CLLocationCoordinate2D
    var trips: [TripDocument]
}

func clusterTripsByProximity(_ trips: [TripDocument], radiusKm: Double = 50.0) -> [CityCluster] {
    var clusters: [CityCluster] = []
    for trip in trips {
        guard let coord = trip.coordinate else { continue }
        if let idx = clusters.firstIndex(where: {
            haversineKm($0.centroid, coord) < radiusKm
        }) {
            clusters[idx].trips.append(trip)
            // Recompute centroid as mean of member trip coordinates
        } else {
            clusters.append(CityCluster(cityName: trip.cityName, centroid: coord, trips: [trip]))
        }
    }
    return clusters.sorted { $0.trips.count > $1.trips.count } // Most-visited first
}
```

**Radius threshold:** 50 km is a reasonable default for "same city area" across diverse locations (works for Tokyo metro, Paris metro, ski resorts). This is Claude's Discretion ‚Äî expose as a constant.

**Haversine formula:** Compute from first principles (3 lines of Swift) ‚Äî do not import MapKit for this; `CLLocation.distance(from:)` can also serve: `CLLocation(lat, lon).distance(from: CLLocation(lat2, lon2)) / 1000`.

### Pattern 3: Photo Carousel with TabView Page Style

**What:** Full-width paged carousel using `TabView(selection:)` with `.tabViewStyle(.page(indexDisplayMode: .never))`. One page per city cluster. Each page loads a single representative PHAsset for that city.

**When to use:** Whenever selection state must be bidirectionally bound between city strip and carousel.

### Pattern 4: PHAsset Loading for Carousel (Single Photo Per City)

**What:** For each city cluster, fetch the single best-match PHAsset (first photo by date within cluster's trip date ranges). The PhotoGalleryStrip pattern is reused but simplified ‚Äî we want one `UIImage` not an array of thumbnails.

```swift
// Source: Nomad/Components/PhotoGalleryStrip.swift ‚Äî adapted pattern
// [VERIFIED: existing PhotoGalleryStrip.swift]
func fetchRepresentativePhoto(for cluster: CityCluster) async -> UIImage? {
    guard let earliestTrip = cluster.trips.min(by: { $0.startDate < $1.startDate }) else { return nil }
    let options = PHFetchOptions()
    options.predicate = NSPredicate(
        format: "creationDate >= %@ AND creationDate <= %@",
        earliestTrip.startDate as CVarArg, cluster.trips.max(by: { $0.endDate < $1.endDate })!.endDate as CVarArg
    )
    options.sortDescriptors = [NSSortDescriptor(key: "creationDate", ascending: true)]
    options.fetchLimit = 1
    let assets = PHAsset.fetchAssets(with: .image, options: options)
    guard let asset = assets.firstObject else { return nil }
    // Use full-width target size: screen width √ó 320pt at 3x scale
    let targetSize = CGSize(width: UIScreen.main.bounds.width * 3, height: 960)
    return await withCheckedContinuation { cont in
        var resumed = false
        PHImageManager.default().requestImage(for: asset, targetSize: targetSize,
            contentMode: .aspectFill, options: nil) { image, info in
            let isDegraded = (info?[PHImageResultIsDegradedKey] as? Bool) ?? false
            if !isDegraded && !resumed {
                resumed = true
                cont.resume(returning: image)
            }
        }
    }
}
```

**Authorization:** Must check `PHPhotoLibrary.authorizationStatus(for: .readWrite)` before any fetch ‚Äî same as PhotoGalleryStrip. If denied, show the "Allow photo access in Settings." message per UI-SPEC.

### Pattern 5: WeatherKit Integration

**What:** Fetch current temperature for a city's coordinate using `WeatherService.shared.weather(for:)`. Requires entitlement + capability in Xcode.

**Steps to integrate:**
1. Add `com.apple.developer.weatherkit` to `Nomad.entitlements` [VERIFIED: currently missing]
2. Enable WeatherKit capability in Xcode ‚Üí Signing & Capabilities
3. Import WeatherKit framework
4. Fetch with:

```swift
// Source: [CITED: developer.apple.com/documentation/weatherkit]
import WeatherKit
import CoreLocation

func fetchTemperature(for coordinate: CLLocationCoordinate2D) async -> String? {
    let location = CLLocation(latitude: coordinate.latitude, longitude: coordinate.longitude)
    do {
        let weather = try await WeatherService.shared.weather(for: location)
        let temp = weather.currentWeather.temperature
        // Format: "‚àí8¬∞C" using Measurement<UnitTemperature>
        let formatted = MeasurementFormatter().string(from: temp)
        return formatted
    } catch {
        return nil   // Pill hidden on error per UI-SPEC
    }
}
```

**Fallback strategy (Claude's Discretion):** WeatherKit requires network and a provisioned bundle ID. During development/simulator, it may return errors. The fallback is: if WeatherKit fails, show "‚Äî¬∞C" during loading, then hide the pill entirely if the error is final. There is no `temperature` field on `TripDocument` ‚Äî the historical fallback in CONTEXT.md refers to the fact that trip date + location could be used to construct a historical weather query, but WeatherKit's `DayWeather` historical API requires iOS 16.4+ and a different call signature. **Recommendation:** attempt `WeatherService.shared.weather(for:)` for current conditions; if it fails (no network, entitlement not provisioned on device), hide the pill silently. Do not implement historical weather fetch in this phase ‚Äî it is an over-complication; the UI-SPEC explicitly says hide the pill when unavailable. [VERIFIED: UI-SPEC Temperature states]

### Pattern 6: GlobeView Wiring (Country Tap ‚Üí CountryDetailSheet)

**Current behavior** (from `GlobeView.swift`): `onTapCountry` calls `viewModel.animateToCountry(code:)`, which sets `scrollToTripId` and sets `showProfileSheet = true`. This opens the ProfileSheet, not a country detail view. [VERIFIED: GlobeView.swift line 235‚Äì237]

**Required change:** Add `showCountryDetail: Bool` and `selectedCountryCode: String?` to `GlobeViewModel`. When `onTapCountry` fires, set these instead of (or in addition to) `showProfileSheet`. Present `CountryDetailSheet` via a new `.sheet` modifier on `GlobeMapView` ‚Äî nested at the same level as the existing ProfileSheet `.sheet`.

```swift
// GlobeView.swift change ‚Äî add alongside existing .sheet(isPresented: $viewModel.showProfileSheet):
.sheet(isPresented: $viewModel.showCountryDetail) {
    if let code = viewModel.selectedCountryCode {
        CountryDetailSheet(
            countryCode: code,
            trips: viewModel.trips.filter { $0.visitedCountryCodes.contains(code) }

```

**Important:** On iOS, a view can only present one `.sheet` at a time. The existing ProfileSheet `.sheet` and the new CountryDetailSheet `.sheet` are siblings on `GlobeMapView` ‚Äî iOS 16+ supports multiple `.sheet` modifiers on the same view, but only one can be `isPresented: true` at a time. When country is tapped, only `showCountryDetail` should be true; `showProfileSheet` should remain false (or be set to false first). [VERIFIED: GlobeView.swift existing pattern; ASSUMED: iOS 16+ multiple .sheet modifier behavior ‚Äî verify on device]

### Anti-Patterns to Avoid

- **Sibling sheets presenting simultaneously:** Do not set both `showProfileSheet` and `showCountryDetail` to `true` at the same time. Only one sheet can present.
- **Fetching all trips in CountryDetailSheet init:** Pass the pre-filtered trips array from GlobeViewModel ‚Äî it already has all trips. Do not start a new Firestore fetch inside CountryDetailSheet.
- **Sticky city strip:** The UI-SPEC explicitly says NOT to implement sticky header behavior ‚Äî place the city strip in the scroll flow naturally. Implementing sticky requires `coordinateSpace` tracking and is brittle.
- **Blocking PHAsset load on main thread:** Always use `PHImageManager.requestImage` with async continuation ‚Äî same pattern as PhotoGalleryStrip. Never call it synchronously.
- **WeatherKit without entitlement:** Adding `import WeatherKit` without the entitlement causes a runtime crash, not a compile error. The entitlement setup is a prerequisite step (Wave 0).
- **Passing TripDocument.cityName directly as city display name:** `cityName` is user-entered text ("Name Your Trip" alert). It is not a geocoded city name. For the city label in the strip and the location block, use `trip.cityName` as-is ‚Äî it is the best available label. Do not attempt reverse geocoding on every trip load.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Temperature fetch | Custom REST weather API client | `WeatherKit.WeatherService.shared.weather(for:)` | Free, privacy-respecting, Apple-native, auto-handles locale/units |
| Photo authorization check | Custom permission flow | `PHPhotoLibrary.authorizationStatus` + existing PhotoGalleryStrip pattern | Pattern already proven in codebase |
| Full-width paged carousel | UIScrollView with manual paging | `TabView(selection:).tabViewStyle(.page(...))` | SwiftUI-native, bidirectional binding, correct page snapping |
| Distance between coordinates | Custom spherical geometry | `CLLocation.distance(from:)` / `haversineKm` (3-line helper) | CoreLocation built-in; no library needed |
| Stats aggregation | Firestore queries per city | In-memory aggregation over `cluster.trips` already in memory | Data is already loaded in GlobeViewModel.trips |

**Key insight:** All trip data needed for stats, trip log list, and photo matching is already in memory in `GlobeViewModel.trips`. No new Firestore reads are needed in CountryDetailSheet ‚Äî this is a pure presentation-layer feature over already-fetched data.

## Common Pitfalls

### Pitfall 1: WeatherKit Entitlement Missing
**What goes wrong:** Build succeeds; app crashes at runtime with `WeatherError.permissionDenied` or similar when `WeatherService.shared.weather(for:)` is called.
**Why it happens:** WeatherKit requires `com.apple.developer.weatherkit` in the `.entitlements` file AND the capability enabled in Xcode. The entitlement is currently absent from `Nomad.entitlements`. [VERIFIED: Nomad.entitlements]
**How to avoid:** Wave 0 task: add entitlement + enable capability in Xcode before writing WeatherKit code.
**Warning signs:** Any WeatherKit call throws immediately without a network request.

### Pitfall 2: Multiple `.sheet` Modifiers Conflict
**What goes wrong:** Tapping a country presents nothing, or the wrong sheet appears, or the app shows a SwiftUI warning ("Attempt to present __ on __ which is already presenting __").
**Why it happens:** If `showCountryDetail` and `showProfileSheet` are both `true` simultaneously, iOS presents only the first sheet modifier's content.
**How to avoid:** When setting `showCountryDetail = true`, ensure `showProfileSheet = false`. The `animateToCountry` method in GlobeViewModel must be updated to set the country state only, not trigger ProfileSheet.
**Warning signs:** Existing ProfileSheet trips appear instead of CountryDetailSheet.

### Pitfall 3: `cityName` Is User-Entered Text, Not Geocoded
**What goes wrong:** City strip shows "My ski trip" or "Saturday adventure" as city labels because these are the user's trip names.
**Why it happens:** `TripDocument.cityName` is set by the user in the "Name Your Trip" alert. It is not a reverse-geocoded city name. [VERIFIED: GlobeView.swift presentTripNameAlert]
**How to avoid:** The city cluster's display name should be the most common `cityName` value among its member trips (or simply `cityName` of the first/representative trip). Accept that labels will be user-authored. The UI-SPEC acknowledges this pattern ‚Äî use it as-is. Do not add a reverse geocoding step.
**Warning signs:** City strip labels vary wildly and don't look like city names.

### Pitfall 4: Photo Authorization Not Re-Checked
**What goes wrong:** Carousel shows blank for users who denied photo access during onboarding, with no indication of why.
**Why it happens:** PHAsset fetch silently returns 0 results when authorization is `.denied` or `.restricted`.
**How to avoid:** Check `PHPhotoLibrary.authorizationStatus(for: .readWrite)` before fetching in `CityPhotoCarousel`. Show the "Allow photo access in Settings." state with "Open Settings" link per UI-SPEC Copywriting.
**Warning signs:** Carousel is always empty; no error shown.

### Pitfall 5: Stats Aggregation Across City Cluster (Distance Accuracy)
**What goes wrong:** "km" stat shows inflated distances because `distanceMeters` is summed across all cluster member trips without filtering.
**Why it happens:** Each `TripDocument.distanceMeters` is the total distance for that entire trip. Multiple trips in the same city cluster simply need their `distanceMeters` summed.
**How to avoid:** Stats are: `logs = cluster.trips.count`, `km = cluster.trips.reduce(0) { $0 + $1.distanceMeters } / 1000`, `photos = count from PHAsset fetch for cluster date range`. This is the correct interpretation ‚Äî "total km traveled across all trips to this city."
**Warning signs:** km value seems 10-100x too high (means you're using raw meters, not dividing by 1000).

### Pitfall 6: TabView Page Animation Conflicts with Bidirectional Binding
**What goes wrong:** Swiping the carousel updates `selectedCityIndex`, which triggers city strip to update its selection indicator, which (if not careful) re-renders the TabView and resets its page position.
**Why it happens:** Writing back to `selectedCityIndex` from `TabView`'s `onChange` and also reading it as the `selection` binding can cause a feedback loop if not structured carefully.
**How to avoid:** Use `TabView(selection: $selectedCityIndex)` directly. SwiftUI's `TabView` with `Binding<Int>` selection handles bidirectional updates correctly ‚Äî swiping a page updates the binding without re-triggering a page jump. Do not add `onChange(of: selectedCityIndex)` that also sets `selectedCityIndex`.
**Warning signs:** Carousel "snaps back" when swiped, or city strip selection indicator lags behind carousel position.

struct CityCluster: Identifiable {

func haversineKm(_ a: CLLocationCoordinate2D, _ b: CLLocationCoordinate2D) -> Double {
    CLLocation(latitude: a.latitude, longitude: a.longitude)

### CountryDetailSheet ViewModel

```swift
// Source: Pattern derived from GlobeViewModel (@Observable) [VERIFIED: GlobeViewModel.swift]
@Observable
@MainActor
final class CountryDetailViewModel {
    let countryCode: String
    let allCountryTrips: [TripDocument]

    var clusters: [CityCluster] = []
    var selectedCityIndex: Int = 0
    var isLoading = true

    // Per-city resolved state
    var photos: [UUID: UIImage] = [:]          // keyed by CityCluster.id
    var temperatures: [UUID: String?] = [:]    // keyed by CityCluster.id; nil = hide pill

    init(countryCode: String, trips: [TripDocument]) {
        self.countryCode = countryCode
        self.allCountryTrips = trips
    }

    func load() async {
        clusters = clusterTripsByProximity(allCountryTrips)
        isLoading = false
        // Kick off photo + weather loads per cluster concurrently
        await withTaskGroup(of: Void.self) { group in
            for cluster in clusters {
                group.addTask { await self.loadCluster(cluster) }
            }
        }
    }

    private func loadCluster(_ cluster: CityCluster) async {
        async let photo = fetchRepresentativePhoto(for: cluster)
        async let temp = fetchTemperature(for: cluster.centroid)
        photos[cluster.id] = await photo
        temperatures[cluster.id] = await temp
    }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `UIPageViewController` for carousels | `TabView(.page)` in SwiftUI | iOS 14+ | Bidirectional `selection` binding is idiomatic; no UIKit bridge needed |
| Custom weather REST clients | WeatherKit `WeatherService` | iOS 16 (WWDC 2022) | Apple-managed, privacy-respecting, no API key; entitlement required |
| Explicit `PHImageRequestOptions.isSynchronous = true` | Async continuation wrapper | iOS 15+ | Non-blocking; avoids main thread stall |
| `sheet(item:)` for optional presentation | `sheet(isPresented:) { if let x = ... }` | Either works | Current codebase uses `isPresented` + optional ‚Äî consistent with existing pattern |

**Deprecated / outdated:**
- `UIKit UIPageControl` for page indicators: replaced by `.tabViewStyle(.page)` index display or custom dot indicators in SwiftUI
- `WKWebView` weather embeds: no, use WeatherKit
- SceneKit globe: already noted as soft-deprecated WWDC 2025 ‚Äî not relevant here but noted in STATE.md

## Open Questions

1. **Does tapping a country dismiss the JourneyPill and ProfileSheet, or do they coexist?**
   - What we know: `GlobeView` hides JourneyPill when `showProfileSheet = true`; both `showProfileSheet` and `showCountryDetail` can be separate booleans
   - What's unclear: UX intent ‚Äî when country detail is shown, should JourneyPill be hidden too? Should ProfileSheet be closed first?
   - Recommendation: When country detail opens, set `showProfileSheet = false` and `showCountryDetail = true`. JourneyPill hides when any sheet is present (already tied to `!viewModel.showProfileSheet` ‚Äî update condition to `!viewModel.showProfileSheet && !viewModel.showCountryDetail`).

2. **Photo count for the stats pill ‚Äî is this PHAsset count or a stored field?**
   - What we know: `TripDocument` has no `photoCount` field; `PHPhotoLibrary` can return count for a date range; PhotoGalleryStrip already fetches this
   - What's unclear: Whether to fetch PHAsset count per cluster (requires a PHFetch per cluster on sheet open) or to defer until photo load completes
   - Recommendation: Derive photo count from the same `PHFetch` call used for the carousel image. Async ‚Äî show "‚Äî" in pill until resolved, then update.

3. **What happens when GlobeView's country tap is fired but ProfileSheet is already open?**
   - What we know: Currently `animateToCountry` sets `showProfileSheet = true` and `scrollToTripId` ‚Äî ProfileSheet was the destination
   - What's unclear: Should a country tap dismiss ProfileSheet and open CountryDetailSheet, or is this scenario impossible (user must dismiss ProfileSheet before tapping globe)?
   - Recommendation: Globe interaction is blocked by the ProfileSheet modal (modal sheets capture all touch events below them). This scenario cannot occur in normal use. But `showProfileSheet` should be set to `false` when `showCountryDetail` is set to `true`, just in case.
