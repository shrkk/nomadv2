<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input -> Firebase Auth | Email/password validated server-side by Firebase |
| User input -> Firestore | Handle written to `usernames` collection via batch write |
| Native permission dialogs | iOS-controlled; app cannot bypass or forge permission grants |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Input Validation | HandleScreen.swift | mitigate | Client-side: enforce lowercase alphanumeric + underscore only, max 30 chars via `.onChange` filtering. Server-side: Firestore rules require auth. |
| T-02-08 | Input Validation | SignUpScreen.swift | mitigate | Firebase Auth validates email format and password strength server-side. Client-side: 8+ character check before enabling CTA. |
| T-02-09 | Information Disclosure | Handle availability check | accept | Authenticated users can probe handle existence; this is by design for the uniqueness UX. Low-sensitivity data. |
| T-02-10 | Denial of Service | Rapid handle availability checks | mitigate | 500ms client-side debounce (D-04) reduces Firestore read volume; Firestore built-in rate limits provide server-side protection. |
</threat_model>
